Why Atheer

Nine things no other mobile inference engine does.

Every mobile inference engine runs LLMs. Here’s why Atheer is the only one built for the NPU era — with predictive thermal management, hierarchical memory, structured output, model-level encryption, prompt injection guardrails, and autonomous agents, all in safe Rust.

The list on the right is deliberately numbered. It is not a feature grid. Each item is a place where a competitor stops, or never starts.

  1. 01

    NPU-first multi-backend

    Auto-probes NPU → GPU → CPU on both iOS and Android — CoreML/ANE, Metal, NNAPI, Vulkan — and selects the fastest available accelerator. Per-operation routing even sends decode to CPU in Eco mode to conserve GPU memory.

    Competitors don’t: no other engine probes NPU before GPU before CPU on both platforms. llama.cpp has no NPU path. MLC LLM has none. ExecuTorch has Qualcomm-only. MLX Swift has none.

  2. 02

    Predictive thermal orchestration

    Least-squares regression on live 1 Hz telemetry predicts temperature trajectory and pre-emptively downgrades inference mode before the OS thermal throttle hits — replacing the sudden 60% performance cliff with graceful degradation.

    No mobile inference engine has thermal-aware operation. Every competitor runs flat-out until the OS forcibly throttles.

  3. 03

    L1/L2/L3 KV cache hierarchy

    A three-tier cache with alignment gating and explicit handoff protocols. L1 holds the active session, L2 retains recent sessions for fast recall, L3 stores compressed/archived context — designed for multi-turn agent sessions on memory-constrained devices. (Note: promotion scoring heuristics are not yet implemented — the alignment score is currently a manually-set placeholder.)

    Every other engine uses a single flat KV cache with no tiering, no promotion, and no cross-session handoff.

  4. 04

    Grammar-constrained decoding

    A first-class GrammarConstraint trait implements a pushdown automaton that guarantees valid JSON output at every decode step — not a post-processing bolt-on, but a native part of the generation loop with thread-safe Send + Sync semantics.

    Only llama.cpp has comparable GBNF grammar support, but it’s a C API bolt-on, not a first-class trait in a safe language.

  5. 05

    Built-in agent loop

    An autonomous Agent struct with configurable max_steps handles the generate → parse → tool-call → continue cycle natively. No need to build orchestration logic in the host app — Atheer ships the loop.

    Every other mobile engine is a text-in-text-out API. You must build agent orchestration, context management, and tool-calling yourself.

  6. 06

    UniFFI cross-platform bindings

    One UniFFI definition generates idiomatic Swift packages for iOS and Kotlin packages for Android — from the same Rust codebase, with no platform-specific wrapper code or duplicate SDK maintenance.

    llama.cpp exposes a raw C API requiring per-platform wrappers. ExecuTorch is C++-only. MLX Swift is Apple-exclusive.

  7. 07

    Safe Rust, not C/C++

    Rust’s ownership model guarantees no use-after-free, no data races, no GC pauses. catch_unwind wraps ANE forward calls so accelerator panics don’t crash the host app. Every line of business logic across inference, grammar, cache, and orchestration is safe Rust — the only unsafe in the entire codebase is at platform FFI boundaries (C ABI export, NDK, Vulkan, ObjC), exactly the same discipline as Servo, wgpu, and Tokio.

    Most mobile inference engines are written in C/C++ with manual memory management across their entire codebase. Only MLX Swift shares Rust’s safety guarantees — but it’s Apple-only.

  8. 08

    Model-level & cache encryption, signature verification, safe load pipeline

    AES-256-GCM encryption for GGUF and .mlpackage model files at rest, with a three-strategy decryption pipeline — ServerDistributed keys, DeviceDerived via HKDF-SHA256, or Custom provider. Ed25519 detached signature verification validates model origin and integrity before loading. A pre-allocation header gate (safe_content::parse_header) runs before any Vec<u8> allocation, rejecting crafted GGUF headers via length-pre-validated metadata scans and bounds-checked counts — with parse_header running before Mmap::map so adversarial sparse files cannot trigger mmap-OOM. Six typed AtheerCoreError variants (InvalidMagic, InvalidCounts, InvalidTensorBounds, …) replace the prior string-only failure mode, with structured fields emitted through the FFI boundary. KV cache snapshots at rest are also encrypted via the same AES-256-GCM primitive (LZ4 compress → encrypt with random 12-byte nonce, distinct AAD b"atheer-cache-v1"), with the encryption key zeroized on drop.

    No other open engine encrypts models, signs models, gates GGUF allocation, or encrypts their KV cache at rest.

  9. 09

    Prompt injection guardrails (L1/L2/L3)

    Three-layer defense-in-depth against prompt injection — L1 fast heuristics (pattern matching, Unicode normalization, leetspeak, homoglyphs, proximity scoring) in <100 μs, L2 token-level statistical analysis (repetition ratio, entropy anomaly, adversarial suffix detection) in <5 ms, and L3 output guard (system prompt leakage detection, jailbreak success markers) in <100 μs. Encoding detection pipeline automatically decodes base64/hex/ROT13 and re-checks decoded text.

    No other mobile inference engine ships built-in guardrails at any layer, let alone a configurable three-tier pipeline with sidecar pattern loading and encoding detection.

Platform support

iOS & Android, every accelerator.

Atheer auto-probes the best available backend in platform-specific priority order. NPU → GPU → CPU. Each backend is a Rust crate module — no black boxes, no per-platform SDK forks.

The five rows on the right show every probe target, in the order they are tried at engine init.

iOS · P1

CoreML / ANE

Apple Neural Engine via candle-coreml integration. ANECompatibility struct with per-layer heuristics, model size ceiling, and quantization whitelist. Background ANE compilation pre-heat via background thread loads and warms up the model at engine init — atomically swapped via Arc<OnceLock> to eliminate cold-start latency. Falls back to Metal for incompatible architectures. catch_unwind protected.

ANE detection · macOS build
iOS · P2

Metal GPU

GPU acceleration via candle_core::Device::Metal. Full tensor ops for prefill and decode in atheer-accel/src/metal.rs.

iOS 15+ · Candle Metal backend
Android · P1

NNAPI NPU/DSP

Raw NDK FFI bindings with ~20 extern functions covering device discovery, graph construction (9 ops), compilation, and execution. Full pipeline in atheer-accel/src/nnapi_ndk.rs.

API 24+ · NDK r29+
Android · P2

Vulkan GPU

Custom GLSL compute shaders (gemv.glsl, attention.glsl) compiled to SPIR-V via naga at build time. DP4A-style int8 matrix-vector multiply and flash attention in atheer-accel/src/vulkan.rs (~1120 lines).

API 26+ · custom compute shaders
Fallback

CPU

Always available fallback using Candle CPU backend. Graceful degradation when no accelerator is available — graceful, tested, documented.

Any platform · zero acceleration
Quick start

From zero to inference.

Build the workspace, run tests, and integrate into your iOS or Android app. Three commands to a running engine, three more to a mobile integration.

The code blocks are real: the bash block is the workspace build, the swift block is iOS, the kotlin block is Android. Each is a copy-paste start, not a fragment.

bash workspace build 
# Build the entire workspace
cargo build --workspace

# Run all 480+ tests
cargo test --workspace

# Download a test model (~350 MB)
scripts/download-test-model.sh

    # Run integration tests with a real model
ATHEER_TEST_MODEL=./models/LFM2-700M-Q4_0.gguf \
  cargo test -p atheer-core

# Benchmark throughput
cargo run -p perf-bench -- \
  --model-path model.gguf \
  --batch-sizes 1,4,8
swift iOS integration 
import AtheerFFI

var config = AtheerConfig()
config.modelPath = "/models/llama.gguf"
config.adaptive = true

let engine = AtheerEngine(config: config)
try engine.initialize()

var request = GenerationRequest(prompt: "Hello")
let response = try engine.generateSync(request: request)
print(response.text)
kotlin Android integration 
import uniffi.atheer_ffi.*

val config = AtheerConfig(

    prompt = "Hello",

    temperature = 0.7f

val response = engine.generateSync(request)
println(response.text)

Roadmap & research

Closing the gaps, shipping the future.

A comprehensive codebase audit across 7 dimensions (security, privacy, reliability, performance, platform, compliance, observability) produced a prioritized 3-phase roadmap. Estimated total: 9–15 weeks for a Rust engineer familiar with the codebase.

Done Phase 1

Security & reliability

2–3 weeks · 7 items
  • Fix 14 test failures + 40+ warnings
  • Fix PII redactor + UTF-8 bugs
  • Un-stub draft speculation
  • Continuous runtime calibration
  • Model loading retry + CPU degrade fallback
  • Sampling thread heartbeat watchdog
  • Model signature + hash verification
Done Phase 2

Security & privacy hardening

3–4 weeks · 10 items
  • Encrypt L2/L3 cache + checkpoints (AES-256-GCM)
  • Model signature + hash verification (S2 + S3)
  • GGUF format validation (S5)
  • Pre-allocation header gate (S6) — safe_content::parse_header rejects crafted GGUF headers before any Vec<u8> allocation; six typed AtheerCoreError variants
  • Configurable privacy mode (PrivacyMode)
  • Prompt injection guardrails (L1/L2/L3)
  • ANE compilation pre-heat (background)
  • KV cache checkpoint persistence
  • Model loading retry + CPU degrade fallback
  • Sampling thread heartbeat watchdog
